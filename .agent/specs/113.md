## Goal

로컬 개발·검증 환경에서 루트 `docker compose` 기준으로 Airflow 3.2.0 + Python 3.13 기반 파이프라인 실행환경을 초기화하고, DAG 파싱·실행·로그 확인·artifact 저장까지 일관되게 재현할 수 있도록 실행환경 구조와 운영 규칙을 정의한다.

이 스펙은 **운영 배포 문서가 아니며**, Spring `pipeline-job` 기능 구현이나 Airflow↔Spring webhook 구현은 범위에서 제외한다.

> **Airflow Version**: `3.2.0`  
> **Python Variant**: `python3.13`  
> **Latest known release at spec time**: `3.2.0` — 로컬 개발 안정성과 Python 3.13 계약 정렬을 위해 의도적으로 `3.2.0`에 고정한다.

## Runtime Topology

### 목적

- 루트 `docker-compose.yml` 기준으로 전체 서비스를 같은 방식으로 띄운다.
- Airflow 3 최소 구성 요소를 빠짐없이 포함한다.
- metadata migration 이전에 Airflow 런타임 서비스가 먼저 뜨지 않도록 one-shot init 규칙을 강제한다.

### 서비스 구성

| Service | 역할 | 필수 여부 | 비고 |
|--------|------|-----------|------|
| `postgres` | 앱 DB + Airflow metadata DB 호스팅 | 필수 | 컨테이너 1개, 단일 DB(`init`) 공유 |
| `backend` | Spring Boot 앱 서버 | 필수 | 기존 서비스 유지 |
| `frontend` | 웹 UI | 필수 | 기존 서비스 유지 |
| `airflow-init` | metadata migration + writable path 확인 | 필수 | one-shot init job |
| `airflow-apiserver` | Airflow UI/API 제공 | 필수 | Airflow 3 기준 `api-server` 사용 |
| `airflow-scheduler` | DAG run 스케줄링/태스크 orchestration | 필수 | `LocalExecutor` |
| `airflow-dag-processor` | DAG 파싱/갱신 | 필수 | Airflow 3 최소 구성 |
| `airflow-triggerer` | deferrable task 이벤트 루프 | 선택 | v1 최소 구성 제외, optional profile |

### 런타임 권장 정책

- `airflow-apiserver`, `airflow-scheduler`, `airflow-dag-processor`에는 `restart: unless-stopped`를 권장한다.
- `airflow-apiserver`에는 healthcheck를 둔다.

```yaml
healthcheck:
  test: ["CMD", "curl", "--fail", "http://localhost:8080/api/v2/version"]
  interval: 30s
  timeout: 10s
  retries: 5
  start_period: 30s
```

### 기동 순서

#### A. Airflow 내부 기동 순서

1. `postgres` healthcheck 통과
2. `airflow-init` 실행 완료
3. `airflow-apiserver`, `airflow-scheduler`, `airflow-dag-processor` 기동

#### B. 애플리케이션 기동 순서

1. `postgres` healthcheck 통과
2. `backend` 기동
3. `frontend` 기동

### Compose 의존 조건

루트 `docker compose` 실행 기준으로 아래 조건을 명시한다.

- `airflow-init`
  - `depends_on.postgres.condition = service_healthy`
- `airflow-apiserver`
  - `depends_on.postgres.condition = service_healthy`
  - `depends_on.airflow-init.condition = service_completed_successfully`
- `airflow-scheduler`
  - `depends_on.postgres.condition = service_healthy`
  - `depends_on.airflow-init.condition = service_completed_successfully`
- `airflow-dag-processor`
  - `depends_on.postgres.condition = service_healthy`
  - `depends_on.airflow-init.condition = service_completed_successfully`
- `backend`
  - `depends_on.postgres.condition = service_healthy`
- `frontend`
  - `depends_on.backend.condition = service_started`

원칙:

- Airflow 런타임 서비스는 migration 완료 전 기동하면 안 된다.
- `airflow-init`는 one-shot init job이다.
- 다른 Airflow 서비스는 `service_completed_successfully` 없이는 시작하지 않는다.
- `backend`와 `frontend`는 Airflow 내부 기동 순서에 종속되지 않는다.

## Bootstrap Flow

### 초기화 시퀀스

```mermaid
sequenceDiagram
    participant dc as docker compose
    participant pg as postgres
    participant ai as airflow-init
    participant ap as airflow-apiserver
    participant sc as airflow-scheduler
    participant dp as airflow-dag-processor

    dc->>pg: start
    pg-->>dc: healthy
    dc->>ai: start
    ai->>pg: connect shared init DB
    ai->>ai: airflow db migrate
    ai->>ai: logs/artifacts/auth writable check
    ai-->>dc: exit 0
    dc->>ap: start
    dc->>sc: start
    dc->>dp: start
    ap->>ap: simple auth password generate
```

### DB 사용 규칙

- Postgres 컨테이너는 1개만 사용한다.
- 앱 데이터와 Airflow metadata는 모두 `init` DB를 사용한다.
- Postgres 접속 계정도 기존 `init` 계정을 그대로 사용한다.
- 앱의 기존 스키마 `app/corpus/pack/review/pipeline/runtime`는 `init` DB 안에 유지한다.
- Airflow metadata 테이블은 같은 `init` DB의 `public` 스키마에 생성된다.

### 초기화 책임

#### `airflow-init`

- `AIRFLOW__DATABASE__SQL_ALCHEMY_CONN`으로 metadata DB 접속
- `airflow db migrate` 수행
- `/opt/airflow/logs`, `/opt/airflow/artifacts`, `/opt/airflow/auth` writable 여부 확인
- 사용자 생성은 수행하지 않음
- 이유: Simple auth manager는 `airflow-apiserver` 기동 시 비밀번호를 자동 생성하므로, init 단계에서 `airflow users create`를 실행하면 인증 방식과 충돌한다.

### 로컬 재부트스트랩 절차

- 완전 초기화: `docker compose down -v`
- DB만 남아 있는 꼬임 복구:
  - `postgres` 볼륨 삭제 후 재기동 또는
  - 기존 `init` DB/계정 기준으로 `airflow-init` 재실행

> ⚠️ `docker compose down -v`는 `airflow_artifacts`, `airflow_logs`, `airflow_auth`를 포함한 **모든 named volume**을 삭제한다. 보존이 필요한 artifact와 로그는 사전에 백업한다.

## Docker Compose And Image Policy

### 표준 명령

- 기본 실행: `docker compose up -d`
- 강제 리빌드가 필요할 때만: `docker compose up --build -d`

이유:

- Compose v2는 로컬에 이미지가 없으면 `up -d`만으로도 자동 빌드를 수행한다.
- `backend`와 `frontend`는 기존 실행환경을 유지하며, 컨테이너 실행 시 선행 산출물(`bootJar`, `pnpm build`)을 전제한다.
- Dockerfile/requirements 변경이 있을 때만 `--build`로 강제 재생성하면 충분하다.

### Airflow 이미지 정책

- 베이스 이미지: `apache/airflow:3.2.0-python3.13`
- 커스텀 이미지는 **항상 이 버전 위에서 확장**
- `ml/airflow/requirements.txt`를 함께 포함하고, `apache-airflow==3.2.0`을 명시해 버전 드리프트를 막는다.
- `ml/airflow/`는 Airflow 런타임 전용 디렉터리로 저장소에 포함한다.

예시 정책:

- `apache-airflow==3.2.0`
- `asyncpg`
- 필요한 provider/client 패키지

금지:

- `_PIP_ADDITIONAL_REQUIREMENTS` 사용
- Airflow 버전 미고정 상태에서 provider만 추가 설치

### Python 버전 계약

- 현재 저장소의 [ml/pyproject.toml](/Users/yeon_seong/coding/ostone/ml/pyproject.toml:1)은 `requires-python = ">=3.13"`을 사용한다.
- Airflow 런타임 이미지는 `python3.13`에 고정한다.
- 따라서 Airflow 런타임 Python과 ML 프로젝트의 최소 Python 계약을 일치시킨다.

이유:

- Airflow 컨테이너는 로컬 `ml` 프로젝트를 패키지로 설치하지 않는다.
- 대신 `./ml/src`를 `/opt/airflow/src`로 bind mount 하고, `PYTHONPATH=/opt/airflow/src`로 모듈을 import 한다.
- 즉 Airflow 런타임은 `pyproject.toml`의 build/install 계약을 직접 사용하지 않고, `ml/src/pipeline/...` 소스 import만 사용한다.
- 그럼에도 런타임 Python을 `>=3.13` 계약과 맞춤으로써, 향후 3.13 전용 문법/stdlib 사용으로 인한 import 실패 가능성을 줄인다.

추가 원칙:

- Airflow 컨테이너 안에서 `pip install -e ./ml` 또는 동등한 방식으로 로컬 `ml` 패키지를 설치하지 않는다.
- 향후 Airflow 이미지에서 `ml` 패키지 설치가 필요해져도, 현재 스펙 기준 Python 버전 계약 충돌은 없다.

## Environment Contract

### 필수 prerequisites

- Docker Engine 설치
- Docker Compose v2 사용
- Docker 메모리 최소 4GB, 권장 8GB

### 호스트별 주의사항

- Linux: 필요하면 `.env`에 `AIRFLOW_UID=$(id -u)`를 넣어 권한 문제를 줄인다.
- macOS / Windows: 기본값 사용 가능
- SELinux / AppArmor의 `:z` 마운트는 기본 스펙이 아니라 트러블슈팅 항목으로 둔다.

### Airflow 런타임 env

| Key | Value / Example | 설명 |
|-----|-----------------|------|
| `AIRFLOW__CORE__EXECUTOR` | `LocalExecutor` | 실행기 고정 |
| `AIRFLOW__DATABASE__SQL_ALCHEMY_CONN` | `postgresql+psycopg2://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}` | sync metadata DB 연결 |
| `AIRFLOW__DATABASE__SQL_ALCHEMY_CONN_ASYNC` | `postgresql+asyncpg://${DB_USER}:${DB_PASSWORD}@postgres:5432/${DB_NAME}` | async metadata DB 연결 |
| `AIRFLOW__CORE__FERNET_KEY` | `compose fallback 또는 override` | 암호화 키 |
| `AIRFLOW__API__SECRET_KEY` | `compose fallback 또는 override` | API secret key |
| `AIRFLOW__API__BASE_URL` | `http://airflow-apiserver:8080` | 내부 API base URL |
| `AIRFLOW__CORE__EXECUTION_API_SERVER_URL` | `http://airflow-apiserver:8080/execution/` | execution API URL |
| `AIRFLOW__API_AUTH__JWT_SECRET` | `compose fallback 또는 override` | Scheduler/API Server 공통 JWT secret |
| `AIRFLOW__CORE__LOAD_EXAMPLES` | `False` | 예제 DAG 비활성화 |
| `AIRFLOW__CORE__SIMPLE_AUTH_MANAGER_USERS` | `admin:admin,viewer:viewer` | Simple auth 사용자 선언 (형식: `username:role`, 비밀번호는 자동 생성됨) |
| `AIRFLOW__CORE__SIMPLE_AUTH_MANAGER_PASSWORDS_FILE` | `/opt/airflow/auth/simple_auth_manager_passwords.json.generated` | 자동 생성 비밀번호 저장 파일 |
| `AIRFLOW__CORE__DAGS_FOLDER` | `/opt/airflow/src/dags` | DAG 디렉터리 |
| `AIRFLOW__LOGGING__BASE_LOG_FOLDER` | `/opt/airflow/logs` | 로그 디렉터리 |
| `PYTHONPATH` | `/opt/airflow/src` | 파이프라인 코드 import 경로 |

`AIRFLOW__SCHEDULER__STANDALONE_DAG_PROCESSOR`는 명시하지 않는다.

- Airflow 3에서는 `dag-processor`가 독립 서비스로 동작하는 구성이 기본 전제다.
- 따라서 이번 스펙은 `airflow-dag-processor` 서비스 자체를 필수로 두고, 2.x 호환 목적의 추가 env는 넣지 않는다.

### 앱/파이프라인 env

| Key | Value / Example | 설명 |
|-----|-----------------|------|
| `DB_NAME` | `init` | 앱 DB 이름 |
| `DB_USER` | `init` | 앱 DB 계정 |
| `DB_PASSWORD` | `<required>` | 앱 DB 비밀번호 |
| `AIRFLOW_UID` | `501` 등 | Linux host 권한 문제 방지 |

로컬 개발에서는 위 값들 대부분을 compose fallback으로 제공하고, 필요할 때만 루트 `.env`로 override 한다.

### 인증 규칙

- 인증은 Simple auth manager로 고정한다.
- 사용하지 않는 방식:
  - `airflow users create`
  - `AIRFLOW_ADMIN_*`
- 비밀번호는 `airflow-apiserver` 시작 시 자동 생성된다.
- 확인 위치:
  - `airflow-apiserver` 로그
  - `/opt/airflow/auth/simple_auth_manager_passwords.json.generated`

## Volumes And Import Rules

### 마운트 규칙

- `./ml/src:/opt/airflow/src`
- `airflow_logs:/opt/airflow/logs`
- `airflow_artifacts:/opt/airflow/artifacts`
- `airflow_auth:/opt/airflow/auth`

주의:

- `ml/src/`는 compose에서 직접 bind mount 한다.
- Airflow는 `AIRFLOW__CORE__DAGS_FOLDER=/opt/airflow/src/dags`로 runtime DAG를 읽는다.
- `ml/tests/dags/`는 테스트 전용이므로 기본 compose mount 대상이 아니다.

### Python import 규칙

- `PYTHONPATH=/opt/airflow/src`
- DAG에서는 `from pipeline.stages...` 사용
- relative import 금지
- DAG에서 import하는 모든 패키지 디렉터리에는 `__init__.py`가 있어야 한다.

### 현재 저장소 확인 사항

- `ml/src/dags/` 존재
- `ml/airflow/` 존재
- `ml/airflow/requirements.txt` 존재
- `ml/src/pipeline/__init__.py` 존재
- `ml/src/pipeline/common/__init__.py` 존재
- 각 stage 디렉터리의 `__init__.py` 존재

검증 근거:

- 로컬 확인 명령: `find ml/src/pipeline -maxdepth 3 \( -type f -o -type d \) | sort`
- 로컬 확인 명령: `test -f ml/airflow/requirements.txt && echo yes-airflow-requirements`
- 로컬 확인 명령:
  `test -f ml/src/pipeline/__init__.py && echo yes-pipeline-init; test -f ml/src/pipeline/common/__init__.py && echo yes-common-init; find ml/src/pipeline/stages -mindepth 1 -maxdepth 1 -type d -exec sh -c 'for d; do if [ -f "$d/__init__.py" ]; then echo yes-stage-init:$d; else echo no-stage-init:$d; fi; done' sh {} + | sort`

따라서 현재 기준으로 `ml/src/pipeline/...` import 구조와 Airflow runtime 디렉터리 구조를 함께 사용한다.

추가 검증 규칙:

- 스펙 문서나 CI에서 실제 경로를 참조할 때는 `test -e <path>` 수준의 존재 검증을 먼저 수행한다.
- 예: `ml/airflow/requirements.txt`처럼 버전 정책에 직접 언급하는 파일은, 문서 생성/검토 단계에서 실제 존재 여부를 확인한 뒤에만 명시한다.

## Artifact-First Contract

### 원칙

- stage 간 데이터 전달은 XCom 중심이 아니라 artifact-first로 설계한다.
- XCom에는 경량 제어 정보만 넣는다.

### v1 최소 artifact 저장소

- 별도 object storage는 도입하지 않는다.
- named volume 기반 로컬 artifact 저장소를 사용한다.
- artifact 루트: `/opt/airflow/artifacts`

### 경로 규칙

```text
/opt/airflow/artifacts/{dag_id}/{run_id}/{stage_name}/
```

### XCom 전달 범위

- artifact manifest 경로
- 요약 메타데이터
- 다음 stage 제어 정보

`pipeline.pipeline_artifact` 등 서비스 DB 기록은 후속 구현 범위로 넘긴다.

## DAG Runtime Defaults

개발 기본값은 아래로 고정한다.

| 설정 | 값 |
|------|----|
| `retries` | `1` |
| `retry_delay` | `timedelta(minutes=5)` |
| `execution_timeout` | `timedelta(minutes=30)` |
| `dagrun_timeout` | `timedelta(hours=2)` |
| `max_active_runs` | `1` |
| `catchup` | `False` |

이유:

- 로컬 개발에서는 무한 재시도보다 빠른 실패 확인이 우선
- 동일 DAG 중복 실행 충돌을 줄임
- 너무 오래 걸리는 태스크를 조기에 식별

## Airflow UI And Runtime Observability

Airflow UI에서 확인 가능해야 하는 항목:

- DAG import 오류
- DAG 파싱 반영 여부
- stage별 성공/실패
- retry 상태
- timeout 상태
- task duration
- task logs
- 최근 DAG run 이력

추가 로그 확인 대상:

- `airflow-dag-processor` 로그
- `airflow-scheduler` 로그
- `airflow-apiserver` 로그

실행 API 진단 포인트:

- `airflow-scheduler`가 `http://airflow-apiserver:8080/execution/`에 도달 가능한가
- `AIRFLOW__API_AUTH__JWT_SECRET`가 모든 Airflow 서비스에 동일하게 주입되는가

태스크 로그 공통 필드 권장:

- `workspace_id`
- `dataset_id`
- `pipeline_job_id`
- `stage_name`
- `artifact_root`

한계:

- Airflow UI는 orchestration 상태와 task 로그 확인에는 충분하다.
- backend 반영 성공 여부나 `pipeline.*` 테이블 write 성공 여부는 이번 범위에서 직접 보장하지 않는다.

## Test Plan

### Acceptance Criteria

- 첫 실행에서 `docker compose up -d` 한 번으로 전체 서비스가 기동된다.
- Dockerfile 또는 의존성 변경 후 `docker compose up --build -d`로 강제 리빌드가 가능하다.
- `airflow-init`가 먼저 성공 종료한 뒤 `airflow-apiserver`, `airflow-scheduler`, `airflow-dag-processor`가 기동된다.
- migration 이전에 Airflow 런타임 서비스가 먼저 뜨지 않는다.
- Airflow UI 접속이 가능하다.
- `admin` 사용자의 자동 생성 비밀번호가 로그 또는 passwords file에서 확인된다.
- 주 실행 DAG가 import error 없이 표시된다.
- DAG 수정 후 `dag-processor`를 통해 UI 반영이 갱신된다.
- `domain_pack_generation` DAG 1회 실행이 성공한다.
- failure-test DAG 1회 실행에서 `retry` / `failed` 상태가 UI에 표시된다.
- execution API 통신 오류 없이 task가 시작된다.
- `/opt/airflow/artifacts/...` 경로가 생성되고 manifest가 남는다.
- Linux host에서 `AIRFLOW_UID` 적용 시 마운트 경로가 root 소유 문제 없이 동작한다.
- `AIRFLOW__DATABASE__SQL_ALCHEMY_CONN_ASYNC`를 명시한 상태에서 metadata DB 연결이 정상 동작한다.

### 검증용 보조 DAG

아래 파일은 검증용 테스트 DAG다. 기본 Airflow compose mount 대상에는 포함하지 않는다.

  - artifact 경로 생성 확인
- `ml/tests/dags/dev_replay.py`

  - retry / failed 상태 확인
