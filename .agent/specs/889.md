# 889 · domain candidate 생성 근거와 fallback 품질 개선

> Issue: feat(ml) domain candidate 생성 근거와 fallback 품질을 개선한다
> Area: ML(주) + 프론트엔드 (mixed). ML 파이프라인 스테이지 템플릿을 기준으로 하고 FE 섹션을 분리한다.

---

## Goal

첫 domain confirmation checkpoint에서 운영자가 후보를 판단할 수 있도록 각 후보에 **표시 가능한
evidence(대표 conversation snippet)** 와 **operator-facing rationale** 을 붙이고, fallback 상황을
원인별로 구분해 artifact와 UI에 드러낸다. `혼합 또는 미확정 0%` 후보가 정상 후보와 섞여 단독 선택지처럼
보이는 상황을 줄이고, domain lexicon의 stopword·일반어·후보 간 중복을 정리한다.

품질이 낮은 후보가 사실상 유일한 선택지처럼 보이면 이후 `intent_discovery -> flow_splitting ->
draft_generation` replay 전체가 잘못된 전제 위에서 진행된다. 이 작업은 domain feedback loop의 시작점
품질을 올린다(#884 confirmed profile 편집과 상호 보완).

---

## Scope

- ML `domain_candidate_generation/main.py`
  - 후보별 `evidenceSnippets`(샘플 대화에서 추출한 표시 가능한 snippet) 추가.
  - 후보별 `rationale`(operator-facing 한 줄 근거) 추가 — LLM 요청 + fallback 합성.
  - fallback을 원인별로 세분화: `llm_request_failure`, `schema_validation_failure`,
    `insufficient_evidence`, `genuinely_mixed`. artifact 최상위 `fallbackReason`과
    `혼합 또는 미확정` 후보 metadata 양쪽에 노출.
  - 후보 `kind` metadata: 정상 후보 `"domain"`, fallback 후보 `"fallback"`(+`isFallback: true`).
  - domain lexicon / evidenceTerms 품질 검증: stopword 제거, 너무 짧은 토큰 제거, 후보 간 과다 중복
    (일반어) term 제거.
- 프론트엔드 `features/pipeline-review`
  - 후보 카드에 rationale와 evidence snippet 표시.
  - fallback 후보를 정상 후보와 시각적으로 구분(badge)하고 `fallbackReason`을 한국어로 설명.
  - 선택한 후보가 fallback이거나 confidence가 낮으면 profile 편집/재검토를 유도하는 안내 표시.

## Non-goals

- 백엔드 변경. `PipelineReviewTaskFactory`/`PipelineReviewCheckpointUseCase`는 candidate row JSON을
  `jsonSupport.toJson(row)` → `readJson` 으로 그대로 통과시키므로 신규 후보 필드는 별도 백엔드 작업 없이
  task payload로 흐른다(하위호환). DTO 화이트리스트가 없음을 확인했다.
- orval codegen. 백엔드 OpenAPI 계약이 바뀌지 않고 후보 payload는 loose object이므로 생성 클라이언트
  재생성이 필요 없다. FE 측 `ReviewTaskPayload` wrapper 타입에만 필드를 추가한다.
- `intent_discovery`/replay 소비 로직 변경. 신규 필드(`rationale`/`evidenceSnippets`/`kind`)는 표시용
  metadata이며 replay 입력은 #884에서 정한 대로 `domainLexicon`/`evidenceTerms`를 계속 사용한다.
- domain candidate 자동 구조 추천·재클러스터링 같은 신규 알고리즘. 본 이슈는 근거 노출과 fallback 분류
  품질을 올리는 작업이다.

## Data / API impact

### `domain-candidates.v1` artifact (필드 추가, 하위호환)

`schemaVersion`은 `domain-candidates.v1` 유지. 기존 필드는 그대로 두고 아래를 **추가**한다.

후보(candidate) 객체 신규 필드:

```json
{
  "candidateId": "card",
  "displayName": "카드 상담",
  "confidence": 0.92,
  "description": "카드 분실·결제·한도 문의",
  "rationale": "카드 분실/정지/한도 문의가 반복적으로 나타납니다.",
  "kind": "domain",
  "evidenceTerms": ["분실", "한도"],
  "evidenceConversationIds": ["c1", "c2"],
  "evidenceSnippets": [
    { "conversationId": "c1", "snippet": "카드를 분실했어요 정지 부탁드립니다" }
  ],
  "suggestedDomainLexicon": ["카드", "결제", "한도"]
}
```

`혼합 또는 미확정` 후보:

```json
{
  "candidateId": "mixed_or_unknown",
  "displayName": "혼합 또는 미확정",
  "confidence": 0.0,
  "kind": "fallback",
  "isFallback": true,
  "fallbackReason": "insufficient_evidence",
  "rationale": "도메인을 분리할 만한 충분한 근거를 찾지 못했습니다.",
  "evidenceSnippets": [ ... ]
}
```

artifact 최상위 신규 필드: `fallbackReason`(정상 후보가 하나도 없을 때/LLM 실패 시에만 non-null,
그 외 `null`). 기존 `generationError`는 유지하고 분류 근거로 사용한다.

`fallbackReason` 분류 규칙:
- `httpx.HTTPError` → `llm_request_failure`
- JSON/스키마 파싱 실패(`ValueError`/`KeyError`/`TypeError`) → `schema_validation_failure`
- 예외는 없으나 정상 후보 0개 + 샘플 수 < `MIN_CANDIDATES` → `insufficient_evidence`
- 예외는 없으나 정상 후보 0개 + 샘플은 충분 → `genuinely_mixed`
- 정상 후보 ≥ 1개 → `null`(이때 `혼합 또는 미확정`은 안전망 대안일 뿐 실패가 아님)

### domain lexicon 품질 규칙
- `evidenceTerms` / `suggestedDomainLexicon` 항목에서 `STOPWORDS` 및 길이 2 미만 토큰 제거.
- 정상 후보가 2개 이상일 때, lexicon term이 전체 정상 후보의 절반을 초과해 등장하면 일반어로 보고
  각 후보 lexicon에서 제거. 단, 후보 lexicon이 비지 않도록 모두 제거될 상황이면 원본을 유지한다.

### 프론트엔드 payload(`ReviewTaskPayload`) 신규 필드(모두 optional)
`rationale?`, `kind?`, `isFallback?`, `fallbackReason?`, `evidenceConversationIds?: string[]`,
`evidenceSnippets?: Array<{ conversationId?: string; snippet?: string }>`.

## Acceptance criteria

1. 정상 domain 후보가 1개 이상이면 선택지로 제공하고, `혼합 또는 미확정` 대안 후보를 항상 함께 둔다
   (이때 분류 규칙대로 `fallbackReason`은 `null`). 정상 후보가 0개면 artifact `fallbackReason`과
   fallback 후보 metadata에 명확한 원인이 채워진다(완료조건 1).
2. 각 후보에 `evidenceConversationIds`와 표시 가능한 `evidenceSnippets`(또는 evidenceTerms)가
   포함된다(완료조건 2).
3. LLM 요청 실패/스키마 실패 시 artifact `fallbackReason`과 후보 metadata에 원인이 드러나고, UI에
   해당 원인이 한국어로 표시된다(완료조건 3).
4. 정상 후보가 있을 때 `혼합 또는 미확정`은 `kind: "fallback"`으로 구분 표시되어 단독 선택지처럼 보이지
   않는다(완료조건 4).
5. artifact는 `domain-candidates.v1`을 유지하고 기존 필드를 보존해 현행 UI/백엔드와 하위호환된다
   (완료조건 5).
6. lexicon/ evidenceTerms에서 stopword·길이 미달·후보 간 과다 중복 term이 제거된다.
7. 후보 카드에 rationale와 evidence snippet이 표시되고, 선택 후보가 fallback이거나 confidence가 낮으면
   profile 편집/재검토 유도 안내가 표시된다.

## Open questions

- `genuinely_mixed`와 `insufficient_evidence`의 경계는 샘플 수 휴리스틱으로만 구분한다. LLM이 명시적으로
  "mixed"를 의도해 반환하는 신호는 현재 프롬프트로 식별 불가하므로 본 이슈에서는 샘플 수 기준으로 근사한다.
