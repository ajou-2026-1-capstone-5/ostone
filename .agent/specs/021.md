## Goal

콘솔 사용자가 워크스페이스를 생성, 조회, 수정, 삭제할 수 있는 CRUD UI를 구현하고 로그인 이후 기본 운영 진입점을 `/workspaces`로 통일하며, 워크스페이스 목록 CRUD와 워크스페이스 범위 운영 진입 화면을 하나의 일관된 정보구조로 재정비한다. 또한 `/workspaces/:workspaceId/workflows`는 해당 workspace의 `publishedAt`이 가장 최근인 domain pack version을 대표 버전으로 사용해 workflow 운영 진입점을 제공한다.

## Design Diff

### As-is vs To-be

| 영역 | As-is | To-be | 변경 내용 |
|------|-------|-------|----------|
| 로그인 이후 기본 진입 | `/upload` 중심 | `/workspaces` 중심 | 운영 시작점 통일 |
| workspace 구조 | 목록 중심 진입점 부재 | 목록 -> 내부 운영 화면 진입 구조 | 정보구조 재정리 |
| workspace 생성 UX | key/name/description 직접 입력 가정 | name만 입력, key 자동 생성 | 생성 장벽 축소 |
| workspace feature 구조 | create/update/archive/list 분산 가정 | `features/workspace` 단일 슬라이스 | FSD 단순화 |
| workspace 카드 | 메타 정보 중심 | 제목/설명/명확한 CTA 중심 고정 카드 | 시인성 및 클릭 의도 강화 |
| 삭제 처리 | soft delete 의미 불명확 | 삭제 후 목록에서 즉시 숨김 | 상태 기반 UX 명확화 |
| workspace 내부 화면 | 별도 빈 홈 페이지 가능성 | `:id` 진입 시 `workflows`로 즉시 redirect | 중간 껍데기 제거 |
| workspace 내비게이션 | 상단/가로형 가정 | 좌측 세로형 `WorkspaceShell` | 운영 섹션 구분 강화 |
| workflow 연결 | workspace와 느슨한 연결 | 최신 published version을 대표 버전으로 선택해 workspace 범위 workflow 진입점 제공 | workspace 기준 운영 동선 명확화 |

## Scope

### In Scope

- `/workspaces` 목록 페이지와 hero/CTA 구조
- workspace 생성, 수정, 삭제 다이얼로그
- `ACTIVE` 상태 workspace만 노출하는 목록 로직
- workspace 카드 고정 크기/고정 레이아웃
- `/workspaces/:workspaceId -> /workspaces/:workspaceId/workflows` redirect
- `WorkspaceShell` 기반 좌측 세로 내비게이션
- workspace 범위 workflow 목록 화면
- 대표 version이 없거나 workflow가 비어 있을 때 upload 화면으로 이동시키는 CTA
- workspace 문맥에서 `UploadPage`를 `WorkspaceShell` 안에서 렌더링하는 구조

### Out of Scope

- consultation 기능 자체 설계 및 workspace-scoped consultation 확정
- `shared/ui` 공용 컴포넌트 API 변경
- login form, global upload form, shared layout 전면 리디자인
- workflow detail 화면(`WorkflowDraftReadPage`) 자체 리디자인
- backend 스키마 변경 또는 `workspaceKey` 제거

## Review Notes

- 예전 021 문서처럼 `workspaceKey` 수동 입력, `create-workspace/update-workspace/archive-workspace` 분리 슬라이스, `/workspaces/:workspaceId` 독립 홈 페이지를 전제로 리뷰하면 오탐이 난다.
- 이번 스펙은 “workspace 목록 CRUD + workspace 범위 운영 진입 구조”를 정의한다. 기존 workflow detail read page는 앱에 존재하더라도 이번 spec에서는 representative version을 거쳐 연결되는 흐름으로 해석한다.
- representative version 규칙은 `publishedAt DESC` 기준 최신 published domain pack version이다. published version이 없으면 workflow empty state를 표시한다.
- 현재 implementation baseline은 placeholder empty state 중심일 수 있다. 리뷰 시에는 baseline과 최종 스펙 목표의 차이를 구분해 보고, latest published version 연결 여부를 후속 반영 포인트로 명시한다.
- 작업 트리에서 workspace 인접 변경이 함께 보일 수 있다. 특히 consultation 연결 변경은 후속 spec 후보이므로, 해당 존재 여부만으로 FE-021의 완료/미완료를 판정하지 않는다.

## API Integration

### Endpoints

이 스펙은 신규 API 정의보다 “기존/동반 구현된 API를 프론트 동선에 맞게 소비하는 방식”을 정리한다.

| Method | Path | 용도 | 소비 위치 |
|--------|------|------|----------|
| GET | `/api/v1/workspaces` | workspace 목록 조회 | `WorkspaceListPage` |
| GET | `/api/v1/workspaces/{workspaceId}` | workspace 단건 조회 | `WorkspaceWorkflowsPage` |
| GET | `/api/v1/workspaces/{workspaceId}/domain-packs/{packId}/versions/{versionId}/workflows` | 대표 version의 workflow 목록 조회 | `WorkspaceWorkflowsPage` |
| POST | `/api/v1/workspaces` | workspace 생성 | `CreateWorkspaceDialog` |
| PATCH | `/api/v1/workspaces/{workspaceId}` | workspace 수정 | `EditWorkspaceDialog` |
| DELETE | `/api/v1/workspaces/{workspaceId}` | workspace soft delete | `ArchiveConfirmDialog` |

### Representative Version Rule

- `/workspaces/:workspaceId/workflows`는 해당 workspace의 domain pack versions 중 `publishedAt`이 가장 최근인 published version을 대표 버전으로 사용한다.
- 대표 버전이 없으면 workflow 목록 대신 empty state를 표시한다.
- 대표 버전 해소를 위한 메타데이터 조회 계약은 동반 구현 또는 후속 API로 보강될 수 있으나, 본 스펙의 핵심 규칙은 `publishedAt DESC` 우선순위다.

### Request / Response 타입

```typescript
export type WorkspaceStatus = "ACTIVE" | "ARCHIVED";
export type WorkspaceMemberRole = "OWNER" | "ADMIN" | "REVIEWER" | "OPERATOR";

export interface WorkspaceResponse {
  id: number;
  workspaceKey: string;
  name: string;
  description: string | null;
  status: WorkspaceStatus;
  myRole: WorkspaceMemberRole;
  createdAt: string;
  updatedAt: string;
}

export interface CreateWorkspaceRequest {
  workspaceKey: string;
  name: string;
  description?: string;
}

export interface UpdateWorkspaceRequest {
  name?: string;
  description?: string | null;
}

```

### Error Mapping

| 코드 | UI 처리 |
|------|---------|
| `WORKSPACE_KEY_CONFLICT` | 생성 실패 메시지 표시 |
| `WORKSPACE_INVALID_KEY` | 자동 생성 key와 서버 규칙이 충돌할 때 에러 표시 |
| `WORKSPACE_INVALID_NAME` | 이름 필드 인라인 에러 우선 처리 |
| `WORKSPACE_NOT_FOUND` | 상태 패널 또는 toast |
| `WORKSPACE_ACCESS_DENIED` | 권한 없음 메시지 |
| 기타 네트워크/알 수 없는 오류 | `mapWorkspaceActionError()` 기본 메시지 |

## Routing

```tsx
<Route path="/" element={<Navigate to="/workspaces" replace />} />
<Route path="/workspaces" element={<PrivateRoute><WorkspaceListPage /></PrivateRoute>} />
<Route path="/workspaces/:workspaceId" element={<PrivateRoute><Navigate to="workflows" replace /></PrivateRoute>} />
<Route path="/workspaces/:workspaceId/workflows" element={<PrivateRoute><WorkspaceWorkflowsPage /></PrivateRoute>} />
<Route path="/workspaces/:workspaceId/upload" element={<PrivateRoute><UploadPage /></PrivateRoute>} />
```

### 라우팅 원칙

- `/workspaces/:workspaceId`는 단독 홈 페이지가 아니라 하위 운영 섹션으로 보내는 alias 라우트다.
- workspace 내부 페이지는 `DashboardLayout` + `WorkspaceShell` 조합으로 렌더링한다.
- `workflows` 섹션은 representative version을 해소한 뒤 workflow 목록 또는 empty state를 렌더링한다.
- consultation을 후속으로 workspace 흐름에 편입할 경우 root-level `/consultation`이 아니라 `/workspaces/:workspaceId/consultation` 하위 경로를 사용한다.
- 기존 workflow detail 라우트는 별도 spec 범위이며, 이번 workspace 진입 동선의 직접 구성요소로 다루지 않는다.

## Data Flow

```text
Route: /workspaces
  -> WorkspaceListPage
     -> workspaceApi.list()
     -> ACTIVE 상태만 필터링
     -> WorkspaceList
        -> WorkspaceCard
           -> onOpen / onEdit / onDelete

CreateWorkspaceDialog
  -> validateCreateWorkspaceForm(name, "")
  -> generateWorkspaceKey(name.trim())
  -> workspaceApi.create({ workspaceKey, name })
  -> onSuccess: 목록 refetch

EditWorkspaceDialog
  -> validateUpdateWorkspaceForm(name, "")
  -> workspaceApi.update(workspaceId, { name })
  -> onSuccess: 목록 refetch

ArchiveConfirmDialog
  -> workspaceApi.archive(workspaceId)
  -> onSuccess: 목록 refetch

Route: /workspaces/:workspaceId/workflows
  -> WorkspaceWorkflowsPage
     -> parseRouteId(workspaceId)
     -> workspaceApi.get(workspaceId)
     -> resolveRepresentativeVersion(workspaceId)
        -> latest published version by publishedAt DESC
     -> if no representative version
        -> WorkspaceShell
           -> Empty state + Upload CTA
     -> workflowApi.list(workspaceId, packId, versionId)
     -> WorkspaceShell
        -> Workflow cards or Empty state

Route: /workspaces/:workspaceId/upload
  -> UploadPage
     -> WorkspaceShell (workspaceId가 있으면)
     -> LogUploadForm(workspaceId)
```

## State Management

### Server State

- TanStack Query는 도입하지 않는다.
- `WorkspaceListPage`, `WorkspaceWorkflowsPage` 모두 `useState + useEffect` 기반으로 로딩/에러/성공 상태를 관리한다.
- mutation 성공 후에는 낙관적 업데이트 대신 전체 목록 재조회로 일관성을 맞춘다.
- `WorkspaceWorkflowsPage`는 representative version 해소 상태와 workflow 목록 상태를 함께 관리한다.

### Client State

- create/edit/delete dialog open state는 `WorkspaceListPage`가 소유한다.
- 선택된 수정/삭제 대상은 `WorkspaceResponse | null` 로컬 state로 관리한다.
- 카드 클릭과 버튼 클릭의 이벤트 전파를 분리해 의도치 않은 이동을 막는다.

## Design Constraints

`frontend/DESIGN.md` 준수:

- 인터페이스 색상은 모노크롬 기반(`#000000` / `#ffffff`)으로 유지한다.
- 타이포그래피는 기존 `figmaSans` 설정을 상속하되, 모노 레이블은 `figmaMono` 계열 uppercase 시각 언어를 따른다.
- CTA와 액션 버튼은 `shared/ui/button.tsx`의 pill geometry를 유지한다.
- focus outline은 기존 `dashed 2px` 패턴을 훼손하지 않는다.
- 카드/다이얼로그 radius는 기존 design token 범위를 따르며, 현재 기준값은 `8px`이다.
- 사용자 피드백은 `alert()` 대신 `toast()`를 사용한다.

### 공유 컴포넌트 활용

| 역할 | `shared/ui` 파일 |
|------|-----------------|
| 생성/수정 다이얼로그 | `dialog.tsx` |
| 삭제 확인 다이얼로그 | `alert-dialog.tsx` |
| 카드 | `card.tsx` |
| 빈 상태 | `empty.tsx` |
| 텍스트 입력 | `input.tsx` |
| 버튼 | `button.tsx` |
| 로딩 표시 | `spinner.tsx` |
| Toast 알림 | `sonner.tsx` |

### 구현 제약

- `shared/ui` 공용 컴포넌트를 직접 수정하지 않고, 조합과 페이지/feature 레벨 스타일로 해결한다.
- workspace spec 범위에서 `features/workspace` 단일 슬라이스를 유지한다.
- `entities/workspace`는 타입/API/도메인 유틸을 책임지고, UI 조합은 상위 계층에서 수행한다.
- `shared -> features/entities` 역방향 import를 만들지 않는다.

## Validation Rules

| 필드 | 클라이언트 검증 | 서버 검증 | 현재 UI 노출 여부 |
|------|---------------|----------|-------------------|
| workspaceKey | 이름 기반 자동 생성 결과를 사용 | 패턴 검증, 중복 체크 | 직접 입력 안 함 |
| name | 필수, 1-255자 | `@NotBlank`, `@Size(max=255)` | 생성/수정에서 사용 |
| description | 선택, max 2000자 | `@Size(max=2000)` | 현재 UI에서는 미노출 |

### 현재 클라이언트 검증 규칙

```typescript
export function validateCreateWorkspaceForm(
  name: string,
  description: string,
): WorkspaceFieldErrors {
  const errors: WorkspaceFieldErrors = {};

  if (!name.trim()) {
    errors.name = "이름을 입력해주세요.";
  } else if (name.length > 255) {
    errors.name = "이름은 255자 이하여야 합니다.";
  }

  if (description.length > 2000) {
    errors.description = "설명은 2000자 이하여야 합니다.";
  }

  return errors;
}
```

## UX / Validation Rules

### Workspace List

- workspace hero와 카드 목록을 시각적으로 분리한다.
- 카드 사이 여백을 충분히 주고 카드 크기는 고정한다.
- 카드 내부 항목은 벽에 붙어 보이지 않도록 내부 padding을 넉넉하게 둔다.
- `workspaceKey`, `myRole`, 최근 업데이트 정보는 카드 본문에서 노출하지 않는다.
- `status !== ACTIVE` 인 workspace는 렌더링하지 않는다.

### Permissions

- 수정 버튼은 `OWNER`, `ADMIN`, 또는 role 값이 비어 있는 경우 노출한다.
- 삭제 버튼은 `OWNER`, 또는 role 값이 비어 있는 경우 노출한다.
- 버튼 클릭은 카드의 open 동작으로 전파되지 않아야 한다.

### Create / Edit Dialog

- 생성 dialog는 이름만 입력받는다.
- 설명 입력 UI는 제공하지 않는다.
- placeholder, 입력 필드, 액션 버튼은 기존 DESIGN.md의 톤과 맞춰 시인성을 확보한다.
- 생성 시 `workspaceKey`는 이름 기반 slug + 랜덤 suffix로 생성한다.
- 이름은 필수이며 255자 이하여야 한다.

### Archive Dialog

- destructive action임을 아이콘, 대비, 경고 박스로 명확히 전달한다.
- 삭제 전 안내는 박스 내부에 배치한다.
- 삭제 성공 후 soft delete 결과가 목록에서 곧바로 반영되어야 한다.

### Workflow List in Workspace

- workflow 목록 화면은 workspace 이름과 현재 섹션을 `WorkspaceShell`에 표시한다.
- representative version은 `publishedAt` 기준 최신 published version으로 결정한다.
- published version이 없으면 empty state를 기본으로 사용한다.
- 대표 version이 있고 workflow가 존재하면 목록을 노출하고, 카드/CTA를 통해 기존 workflow detail read route로 이동할 수 있어야 한다.
- empty state 화면은 업로드로 이어지는 CTA를 제공해야 한다.

## Done Criteria

- [ ] `/` 기본 진입이 `/workspaces`로 통일된다.
- [ ] `/workspaces`에서 로딩/에러/빈 상태/정상 목록을 모두 처리한다.
- [ ] `ACTIVE` 상태 workspace만 렌더링한다.
- [ ] 생성 dialog는 이름만 입력받고 key는 자동 생성해 서버로 보낸다.
- [ ] 수정 dialog는 현재 이름 수정만 지원한다.
- [ ] 삭제 성공 후 해당 workspace는 목록에서 보이지 않는다.
- [ ] workspace 카드 전체 클릭과 Open CTA 모두 동일한 진입 경로를 사용한다.
- [ ] `/workspaces/:workspaceId`는 빈 홈 없이 `workflows`로 redirect 된다.
- [ ] `WorkspaceShell`이 workspace 범위의 최소 운영 섹션을 제공한다.
- [ ] `WorkspaceWorkflowsPage`는 representative version 규칙(`publishedAt DESC`)에 따라 latest published version을 해소한다.
- [ ] representative version이 있으면 해당 version의 workflow 목록을 표시한다.
- [ ] representative version이 없거나 workflow가 비어 있으면 empty state를 표시한다.
- [ ] empty state에서 upload 화면으로 이어지는 CTA가 있다.
- [ ] `shared/ui` 공용 컴포넌트 API를 건드리지 않고 페이지/feature/entity 범위에서 해결한다.
- [ ] FSD 의존성 방향을 위반하지 않는다.

## Follow-up Notes

- `workspaceKey`를 장기적으로 서버 생성 책임으로 넘길지는 backend contract와 별도 합의가 필요하다.
- consultation을 workspace 하위 섹션으로 완전히 편입할지는 후속 spec에서 다루는 편이 안전하며, 경로는 `/workspaces/:workspaceId/consultation`을 기준으로 잡는다.
- representative version 해소를 위한 metadata read contract는 backend/domain-pack 스펙과 함께 구체화할 필요가 있다.
