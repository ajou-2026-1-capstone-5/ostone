## Goal

운영자 로그인 화면이 인증 전 진입점 역할을 맡고, 인증 성공 후에는 기본적으로 `/workspaces`로 이동하며, 보호 라우트에서 튕겨 온 경우에는 안전한 내부 private route로만 복귀하는 규칙을 명확히 정의한다.

## Design Diff

### As-is vs To-be

| 영역 | As-is | To-be | 변경 내용 |
|------|-------|-------|----------|
| 루트 진입 해석 | `/login`이 사실상 첫 화면 | 로그인 성공 후 기본 시작점은 `/workspaces` | 운영 진입점 정렬 |
| 로그인 성공 후 이동 | `/upload` | `/workspaces` 또는 `state.from` | workspace 중심 흐름으로 수정 |
| 보호 라우트 복귀 | `PrivateRoute`가 `state.from` 저장 | 로그인 spec에서 소비 규칙까지 명시 | 문서 간 역할 연결 |

## Data Flow

```text
Route: /login
  -> LoginPage
     -> AuthLayout
     -> LoginForm
        -> validate(email, password)
        -> loginApi({ email, password })
        -> saveAuthSession(tokens, user)
        -> resolvePostLoginDestination(location.state.from)
           -> if from exists: navigate(from, { replace: true })
           -> else: navigate("/workspaces", { replace: true })

PrivateRoute
  -> isAuthenticated()
  -> if unauthenticated
     -> clearAuthSession()
     -> Navigate("/login", { state: { from: location }, replace: true })
```

`resolvePostLoginDestination`은 `state.from`을 그대로 신뢰하지 않고 "Safe Return Destination" 규칙을 먼저 적용한다.

## Routing

### Route Rules

| 상황 | 기대 동작 |
|------|----------|
| 비인증 사용자가 `/login` 진입 | 로그인 화면 표시 |
| 비인증 사용자가 private route 진입 | `PrivateRoute`가 `/login`으로 보내고 `state.from` 저장 |
| 인증 사용자가 `/login` 재진입 | `/workspaces`로 replace 이동 |

### Post-login Destination

1. `location.state.from`이 안전한 내부 private route이면 그 경로를 우선 복귀 대상으로 사용한다.
2. `state.from`이 없거나 안전하지 않으면 기본 진입점인 `/workspaces`로 이동한다.
3. root-level 기본 운영 시작점은 `021.md` 기준 `/workspaces`이며, login spec은 그 전 단계만 책임진다.

### Safe Return Destination

- `state.from.pathname`이 존재하고, 외부 URL이 아니며, 명시적 allowlist prefix 중 하나로 시작할 때만 복귀 대상으로 사용한다.
- 이번 스펙에서 허용하는 return path prefix는 `["/workspaces"]`뿐이다.
- `/login`, `/signup`, `/reset-password`, `/reset-password/complete` 같은 auth route는 allowlist에 포함하지 않으므로 항상 `/workspaces`로 fallback한다.
- pathname이 없거나 allowlist에 포함되지 않은 값은 모두 `/workspaces`로 fallback한다.
- 이 규칙은 open redirect와 auth route 왕복 루프를 피하기 위한 최소 방어선이다.

## Implementation Outline

### App Routing

`App.tsx`는 auth entry와 workspace entry의 책임을 분리한다.

```tsx
<Route path="/" element={<Navigate to="/workspaces" replace />} />
<Route path="/login" element={<LoginPage />} />
<Route path="/signup" element={<SignupPage />} />
<Route path="/reset-password" element={<PasswordResetInitPage />} />
<Route path="/reset-password/complete" element={<PasswordResetCompletePage />} />
<Route path="/workspaces" element={<PrivateRoute><WorkspaceListPage /></PrivateRoute>} />
```

- `/`는 더 이상 `/login`으로 보내지 않고 `/workspaces`로 보낸다.
- `/workspaces` 진입 후 인증이 없으면 `PrivateRoute`가 `/login`으로 보낸다.
- `signup`과 password reset route는 login flow의 보조 auth route로 유지하되, 본 스펙에서 상세 UX를 확장하지 않는다.

### LoginPage

`LoginPage`는 인증된 사용자의 `/login` 재진입을 막고, 비인증 사용자에게만 로그인 폼을 보여준다.

```tsx
export function LoginPage() {
  if (isAuthenticated()) {
    return <Navigate to="/workspaces" replace />;
  }

  return (
    <AuthLayout>
      <LoginForm />
    </AuthLayout>
  );
}
```

- 인증 판별은 access token 유효성 기준으로 한다.
- 인증된 사용자가 로그인 화면을 다시 보는 상태를 허용하지 않는다.

### LoginForm

`LoginForm`은 인증 API 호출, 세션 저장, 안전한 후속 라우팅만 책임진다.

```tsx
const location = useLocation();
const navigate = useNavigate();

const result = await loginApi({ email, password });
saveAuthSession(toAuthTokens(result), result.user);

const destination = resolvePostLoginDestination(location.state);
navigate(destination, { replace: true });
```

- 로그인 성공 후 기본 목적지는 `/workspaces`다.
- `state.from`이 safe return destination이면 해당 경로로 이동한다.
- `state.from`이 auth route이거나 유효하지 않으면 `/workspaces`로 fallback한다.
- 현재 UI에 존재하는 `아이디 저장` checkbox는 실제 저장 로직이 없으므로, 구현 시 제거하거나 별도 remember-id spec으로 분리한다. 본 스펙에서는 remember-id 기능을 정의하지 않는다.

### Safe Destination Helper

`resolvePostLoginDestination`은 login feature 내부 유틸로 둔다. 여러 feature에서 공유하기 전까지 `shared`로 올리지 않는다.

```ts
const DEFAULT_POST_LOGIN_PATH = "/workspaces";
const ALLOWED_RETURN_PREFIXES = ["/workspaces"];

export function resolvePostLoginDestination(state: unknown): string {
  const pathname = extractPathname(state);

  if (!pathname || isExternalUrl(pathname)) {
    return DEFAULT_POST_LOGIN_PATH;
  }

  const isAllowed = ALLOWED_RETURN_PREFIXES.some((prefix) =>
    pathname.startsWith(prefix),
  );

  return isAllowed ? pathname : DEFAULT_POST_LOGIN_PATH;
}
```

- 외부 URL 문자열을 직접 받아 navigate하지 않는다.
- `Location` 객체 형태의 `state.from.pathname`만 후보로 추출한다.
- 추출한 pathname은 `ALLOWED_RETURN_PREFIXES`의 prefix allowlist를 통과해야 한다.
- query string이 필요한 경우에는 `pathname + search` 조합으로 제한하고, hash까지 보존할지는 구현 시 별도 판단한다.

### PrivateRoute

`PrivateRoute`는 user JSON 존재 여부가 아니라 access token 유효성을 기준으로 보호 여부를 판단한다.

```tsx
if (!isAuthenticated()) {
  clearAuthSession();
  return <Navigate to="/login" state={{ from: location }} replace />;
}

return <>{children}</>;
```

- token이 없거나 만료되었거나 파싱에 실패하면 비인증 상태로 본다.
- 유효하지 않은 세션은 즉시 정리해 stale user 정보만으로 private route를 통과하지 못하게 한다.
- current `location`은 로그인 후 복귀를 위해 `state.from`에 저장한다.

## State Management

### Responsibility

- `LoginPage`는 인증 전 로그인 화면을 렌더링하고, 인증된 사용자의 `/login` 재진입을 `/workspaces`로 돌린다.
- `LoginForm`은 이메일 / 비밀번호 입력, 기본 검증, auth API 호출, 세션 저장, 후속 라우팅을 책임진다.
- `PrivateRoute`는 access token 유효성을 기준으로 보호 라우트를 판별하고, 비인증 사용자의 현재 위치를 `state.from`에 저장한다.

### Client State

- `email`
- `password`
- `isLoading`
- `error`

### Session State

- 로그인 성공 시 access token / refresh token / 사용자 정보를 저장한다.
- 이후 private route는 access token 유효성을 기준으로 인증 상태를 판별한다.
- access token이 없거나 만료/파싱 실패 상태이면 저장된 auth session을 정리한다.
