# [FE-011] Console — Login 진입 및 Workspace 랜딩 정렬

> **Backlog**: 비인증 운영자가 로그인 화면에서 인증을 완료한 뒤, workspace 중심 콘솔 흐름으로 안정적으로 진입하고 싶다 → 로그인 이후 기본 운영 시작점을 `/workspaces`로 통일하고 보호 라우트 복귀 규칙을 명확히 하기 위해
> **Layer**: Frontend (Operator Console)
> **Template**: `.agent/specs/_TEMPLATE_FE.md`
> **Branch**: `docs/workspace-flow-spec-alignment`
> **Depends on**: `.agent/specs/021.md`, `frontend/DESIGN.md`
> **Verified existing paths**: `frontend/src/app/App.tsx`, `frontend/src/shared/ui/PrivateRoute.tsx`, `frontend/src/pages/login/ui/LoginPage.tsx`, `frontend/src/features/auth/ui/login-form/LoginForm.tsx`

---

## Goal

운영자 로그인 화면이 인증 전 진입점 역할을 맡고, 인증 성공 후에는 기본적으로 `/workspaces`로 이동하며, 보호 라우트에서 튕겨 온 경우에는 원래 요청한 workspace 하위 경로로 복귀하는 규칙을 명확히 정의한다.

---

## Scope

### In Scope

- `/login` 화면의 역할과 책임
- `POST /api/v1/auth/login` 호출 규칙
- 로그인 성공 후 세션 저장과 후속 라우팅 규칙
- `PrivateRoute`가 저장한 `state.from` 복귀 정책
- 로그인과 workspace 진입 흐름의 연결

### Out of Scope

- 회원가입 / 비밀번호 재설정 상세 UX 개편
- backend auth API contract 변경
- 역할별 대시보드 분기
- consultation 상세 기능 설계

## Design Diff

### As-is vs To-be

| 영역 | As-is | To-be | 변경 내용 |
|------|-------|-------|----------|
| 루트 진입 해석 | `/login`이 사실상 첫 화면 | 로그인 성공 후 기본 시작점은 `/workspaces` | 운영 진입점 정렬 |
| 로그인 성공 후 이동 | `/upload` | `/workspaces` 또는 `state.from` | workspace 중심 흐름으로 수정 |
| 보호 라우트 복귀 | `PrivateRoute`가 `state.from` 저장 | 로그인 spec에서 소비 규칙까지 명시 | 문서 간 역할 연결 |
| workspace와의 관계 | 느슨함 | login은 workspace 흐름의 전단계 | 문서 책임 분리 |

## Route Policy

### 진입 규칙

| 상황 | 기대 동작 |
|------|----------|
| 비인증 사용자가 `/login` 진입 | 로그인 화면 표시 |
| 비인증 사용자가 private route 진입 | `PrivateRoute`가 `/login`으로 보내고 `state.from` 저장 |
| 인증 사용자가 `/login` 재진입 | `/workspaces`로 보내는 것이 바람직함 |

### 로그인 성공 후 이동 규칙

1. `location.state.from`이 존재하면 그 경로를 우선 복귀 대상으로 사용한다.
2. `state.from`이 없으면 기본 진입점인 `/workspaces`로 이동한다.
3. root-level 기본 운영 시작점은 `021.md` 기준 `/workspaces`이며, login spec은 그 전 단계만 책임진다.

### Workspace 하위 경로 연결

- upload: `/workspaces/:workspaceId/upload`
- workflow entry: `/workspaces/:workspaceId/workflows`
- consultation: 이번 작업에서는 상세 spec으로 확장하지 않지만, 후속 편입 시 `/workspaces/:workspaceId/consultation` 하위 경로를 따른다.

## Screen Responsibility

### LoginPage

- 비인증 운영자용 인증 진입 화면을 렌더링한다.
- 시각적 레이아웃은 `AuthLayout`에 위임한다.

### LoginForm

- 이메일 / 비밀번호 입력과 기본 검증을 처리한다.
- auth API를 호출한다.
- 성공 시 세션 저장과 후속 라우팅을 수행한다.
- 실패 시 사용자 메시지를 표시한다.

### PrivateRoute

- 인증 여부를 판별한다.
- 비인증 사용자를 `/login`으로 보낸다.
- 로그인 후 복귀를 위해 현재 위치를 `state.from`에 저장한다.

## API Integration

### Endpoint

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/v1/auth/login` | 운영자 로그인 |

### Request / Response

```ts
export interface LoginRequest {
  email: string;
  password: string;
}

export interface LoginResponse {
  accessToken: string;
  refreshToken: string;
  tokenType: string;
  expiresIn: number;
  user: {
    id: number;
    email: string;
    name: string;
    role: string;
  };
}
```

### Error Mapping

| 에러 코드 | UI 처리 |
|----------|---------|
| `INVALID_CREDENTIALS` | "이메일 또는 비밀번호가 올바르지 않습니다." |
| `PASSWORD_RESET_REQUIRED` | "비밀번호 재설정이 필요합니다." |
| `VALIDATION_ERROR` | 서버 메시지 표시 |
| 기타 API 오류 | 서버 메시지 또는 기본 실패 메시지 |
| 네트워크 오류 | "서버에 연결할 수 없습니다. 잠시 후 다시 시도해주세요." |

## Data Flow

```text
Route: /login
  -> LoginPage
     -> AuthLayout
     -> LoginForm
        -> validate(email, password)
        -> loginApi({ email, password })
        -> saveAuthSession(tokens, user)
        -> resolvePostLoginDestination(location.state.from)
           -> if from exists: navigate(from, { replace: true })
           -> else: navigate("/workspaces", { replace: true })

PrivateRoute
  -> getAuthUser()
  -> if unauthenticated
     -> Navigate("/login", { state: { from: location }, replace: true })
```

## State Management

### Client State

- `email`
- `password`
- `isLoading`
- `error`

### Session State

- 로그인 성공 시 access token / refresh token / 사용자 정보를 저장한다.
- 이후 private route는 저장된 사용자 세션을 기준으로 인증 상태를 판별한다.

## Done Criteria

- [ ] 로그인 화면의 책임이 인증 진입과 후속 라우팅으로 명확히 정의된다.

- [ ] upload / workflow / consultation의 workspace 하위 경로 원칙이 문서에 반영된다.
- [ ] signup / password reset은 이번 범위 밖으로 분리된다.
- [ ] `021.md`와 충돌하지 않는 로그인 진입 규칙을 유지한다.

## Notes

- root-level 기본 운영 랜딩과 workspace 구조는 `021.md`가 최종 책임을 가진다.
- 본 스펙은 auth UX 전체를 재설계하는 문서가 아니라, 로그인 이후 workspace 흐름에 진입하는 연결 규칙을 고정하는 문서다.
