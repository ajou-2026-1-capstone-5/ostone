# CI와 pre-commit 품질 검사 스크립트 단일화

## Goal

루트 `package.json`의 package scripts를 CI, pre-commit, 문서의 공통 품질 검사 진입점으로 사용해 로컬과 PR 검증 기준을 예측 가능하게 만든다.

## Problem

현재 pre-commit은 `lint-staged.config.js`에서 backend Spotless, frontend lint/format, ML ruff/format/mypy를 직접 실행한다. 반면 `.github/workflows/ci.yml`은 backend checkstyle을 제외한 backend/frontend/ML 테스트, coverage gate, 빌드를 직접 실행한다. README와 `.agent/rules/testing.md`도 일부 명령이 실제 CI와 다르게 설명되어 있어, 어떤 검사가 PR을 막고 어떤 검사는 staged 파일 또는 full/manual 검증에서 수행되는지 한눈에 알기 어렵다.

## Scope

- 루트 `package.json`에 backend/frontend/ML 품질 검사, 테스트, CI용 package scripts를 추가한다.
- `lint-staged.config.js`는 직접 도구 명령 대신 루트 package scripts를 호출한다.
- `.github/workflows/ci.yml`의 basic CI, mocked E2E, coverage/Sonar 사전 검사는 루트 package scripts를 호출한다.
- README와 `.agent/rules/testing.md`는 pre-commit, CI basic, CI full/manual의 책임을 같은 기준표로 설명한다.

## Non-goals

- 개별 backend/frontend/ML 테스트 코드 또는 애플리케이션 동작을 변경하지 않는다.
- CI에서 backend checkstyle, frontend lint, ML ruff/mypy를 새 blocking basic job으로 승격하지 않는다.
- SonarCloud 스캔 action, branch protection, coverage threshold 정책을 새로 설계하지 않는다.
- package manager 또는 lockfile 구조를 변경하지 않는다.

## Affected Paths

- `package.json`
- `lint-staged.config.js`
- `.github/workflows/ci.yml`
- `README.md`
- `.agent/rules/testing.md`
- `.agent/specs/621.md`

## Requirements

1. backend/frontend/ML의 반복 품질 명령은 루트 package scripts에 이름 있는 진입점으로 존재해야 한다.
2. pre-commit은 staged 파일 검사 책임을 유지하되, 직접 도구 명령 대신 루트 package scripts를 호출해야 한다.
3. CI basic job은 PR에서 막아야 하는 최소 품질선으로 유지하고, 실행 명령은 루트 package scripts를 호출해야 한다.
4. CI에서 생략되는 검사와 그 이유, 대체 실행 위치가 문서화되어야 한다.
5. frontend lint와 ML ruff/mypy의 실행 위치가 pre-commit 및 full/manual 검증으로 명확히 설명되어야 한다.
6. 루트 package scripts 변경은 관련 CI path filter가 감지할 수 있어야 한다.

## Quality Check Contract

| 구분             | 책임                                  | package scripts                                                                                                                         |
| ---------------- | ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| pre-commit       | 빠른 staged 파일 검사                 | `format:backend:check`, `lint:frontend`, `format:frontend`, `lint:ml`, `format:ml`, `typecheck:ml`                                      |
| CI basic         | PR에서 반드시 막아야 하는 최소 품질선 | `ci:backend`, `ci:frontend`, `e2e:frontend`, `ci:ml`                                                                                    |
| CI full / manual | 느리거나 포괄적인 품질 검사           | `test:backend:coverage`, `ci:backend:sonar`, `test:frontend:coverage`, `test:ml:coverage`, `lint:backend`, `quality:ml`, `format:check` |

## Data/API Impact

- API 계약, database schema, migration, runtime configuration은 변경하지 않는다.
- CI workflow의 실행 명령과 문서화된 개발자 명령만 변경한다.

## Acceptance Criteria

- `pnpm run`으로 backend/frontend/ML의 CI, lint, format, test 명령을 확인할 수 있다.
- `lint-staged.config.js`는 backend/frontend/ML 검사에서 루트 package scripts를 호출한다.
- `.github/workflows/ci.yml`의 backend/frontend/ML basic job은 `pnpm run ci:*` 스크립트를 호출하고, frontend E2E job도 루트 package script를 호출한다.
- README와 `.agent/rules/testing.md`가 pre-commit, CI basic, CI full/manual의 책임을 같은 기준으로 설명한다.
- CI에서 생략하는 backend checkstyle, frontend lint, ML ruff/mypy의 실행 위치와 생략 이유가 문서화된다.

## Validation

- `node -e 'JSON.parse(require("fs").readFileSync("package.json", "utf8"))'`
- `ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci.yml")'`
- `pnpm run format:check`
- `pnpm exec prettier --check .agent/rules/testing.md .agent/specs/621.md`
- 필요 시 모듈별 변경에서 `pnpm run ci:backend`, `pnpm run ci:frontend`, `pnpm run ci:ml`
