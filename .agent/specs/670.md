# 사용자 채팅 메시지 순서 race condition 방지 (seqNo 정렬 도입)

## Goal

사용자(고객) 채팅 화면(`UserChatPage`, `ChatRoom`)에서 사용자 메시지와 챗봇/상담사 응답이 항상 서버가 확정한 순서대로 표시되도록 하여, 거의 동시에 도착하거나 역순으로 도착하는 실시간 이벤트에서도 대화 맥락이 깨지지 않게 한다.

## Problem

사용자 채팅은 공유 모듈 `entities/chat/lib/chatMessageSync.ts`를 통해 REST 스냅샷, WebSocket 실시간 이벤트, optimistic(local) 메시지를 병합/정렬한다. 현재 정렬 기준은 `compareMessageCreatedAt`(= `createdAt` 단독)뿐이며 tie-breaker가 없다. 동일 시각이거나 시계 오차·역순 도착 시 사용자·봇 메시지 순서가 뒤집힐 수 있다. 서버 응답 타입 `ChatMessageResponse`에는 `seqNo`가 있으나 `toChatMessage`가 이를 버려서 정렬에 활용되지 못하고, 실시간 타입 `RealtimeChatMessage`와 도메인 타입 `ChatMessage`에는 `seqNo` 필드 자체가 없다. 운영자 화면(#557)과 동일한 근본 원인이 사용자 채팅 경로에 그대로 남아 있다.

## Scope

- `frontend/src/entities/chat/model/types.ts` (`ChatMessage`에 `seqNo?` 추가)
- `frontend/src/entities/chat/lib/chatMessageSync.ts` (`seqNo` 보존·정렬 강화)
- `frontend/src/entities/chat/lib/chatMessageSync.test.ts` (회귀 테스트 추가)
- `frontend/src/entities/chat/api/chatApi.ts` (`toDemoChatMessage`가 배열 인덱스를 `seqNo`로 보존)
- `frontend/src/entities/chat/api/chatApi.test.ts` (변환 기대값 갱신)

## Non-Goals

- 운영자 상담 화면(`ConsultationPage`)의 메시지 순서 처리 → #557 에서 별도 진행.
- 백엔드의 `seqNo` 채번/발급 로직은 변경하지 않는다(현행 서버 `seqNo`를 신뢰).
- 메시지 페이지네이션(이전 메시지 로드) 및 WebSocket 연결/재연결 동작은 변경하지 않는다.

## Current Contract

| 영역 | 확인된 파일 | 현재 상태 |
| --- | --- | --- |
| 메시지 정렬 | `chatMessageSync.ts` (`compareMessageCreatedAt`, `mergeMessages`) | `createdAt` 단독 정렬, tie-breaker 없음 |
| 메시지 변환 | `chatMessageSync.ts` (`toChatMessage`) | 서버 `seqNo`를 버림 |
| 실시간 수신 | `chatMessageSync.ts` (`RealtimeChatMessage`, `toRealtimeChatMessage`) | `seqNo` 필드 자체가 없음 |
| 도메인 타입 | `types.ts` (`ChatMessage`) | `seqNo` 없음 |

## Design Diff

| 영역 | As-is | To-be |
| --- | --- | --- |
| 정렬 기준 | `createdAt` 단독 | `seqNo` → `createdAt` → 안정 tie-breaker(원래 순서) |
| `seqNo` 보존 | `toChatMessage`가 버림, 실시간 타입엔 필드 없음 | 도메인·실시간 타입과 두 변환 함수 모두 `seqNo` 보존 |
| 병합 안정성 | `Array.sort` 암묵 안정성에 의존 | 원본 인덱스를 명시 tie-breaker로 사용 |
| 데모 워크플로 | 배열 순서에만 암묵 의존 | 응답 인덱스를 `seqNo`로 명시 보존 |

## Requirements

- 사용자 메시지와 챗봇/상담사 메시지는 서버가 확정한 `seqNo` 순서대로 표시된다.
- WebSocket 이벤트가 역순으로 도착해도 화면 순서가 안정적으로 정렬된다.
- optimistic(local) 메시지가 서버 echo와 병합될 때 순서가 뒤틀리지 않는다(`seqNo` 없는 메시지는 서버 확정 메시지 뒤로).
- 동일 `seqNo` 또는 동일 시각 메시지는 원래(병합) 순서를 유지하는 안정 tie-breaker를 가진다.

## Test Plan

- `chatMessageSync.test.ts`: `seqNo` 정렬 / 역순 도착 안정 정렬 / optimistic(seqNo 없음) 후순위 배치 / 동일 `seqNo`·시각 원래 순서 유지 / 실시간 `seqNo` 보존·검증.
- `chatApi.test.ts`: `toChatMessage`·`toDemoChatMessage` 변환 결과에 `seqNo`가 보존되는지 기대값 갱신.
