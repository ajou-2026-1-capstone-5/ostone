# 루트 패키지 매니저 pnpm 단일화

## Goal

루트 Node.js 의존성 관리 정책을 pnpm으로 단일화해 신규 개발자와 CI가 동일한 lockfile과 명령 체계를 사용하도록 한다.

## Problem

루트에는 `package-lock.json`과 `pnpm-lock.yaml`이 함께 존재하고, `frontend/package.json`은 `packageManager`로 pnpm을 명시한다. 반면 루트 `package.json`에는 공식 package manager가 없고 Husky 훅은 `npx`를 사용한다. 이 상태에서는 `npm install`과 `pnpm install` 중 어떤 명령을 써야 하는지 모호하며 lockfile drift가 발생할 수 있다.

## Scope

- `package.json`
- `pnpm-lock.yaml`
- `.husky/pre-commit`
- `.husky/commit-msg`
- `.github/workflows/ci.yml`
- `README.md`
- `AGENTS.md`
- `package-lock.json`

## Non-Goals

- `frontend/` 의존성 버전이나 `frontend/pnpm-lock.yaml`을 변경하지 않는다.
- Backend Gradle, ML uv, Docker Compose 실행 정책을 변경하지 않는다.
- CI job 구조나 paths-filter 정책을 새로 설계하지 않는다.

## Current Contract

| 영역                | 확인된 파일                              | 현재 상태                                                                             |
| ------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------- |
| Root package        | `package.json`                           | `packageManager` 필드 없음, `prepare`/`format` 스크립트 존재                          |
| Root lockfile       | `package-lock.json`, `pnpm-lock.yaml`    | npm/pnpm lockfile이 동시에 존재                                                       |
| Husky hooks         | `.husky/pre-commit`, `.husky/commit-msg` | `npx lint-staged`, `npx commitlint` 사용                                              |
| CI frontend install | `.github/workflows/ci.yml`               | 기본 frontend job은 `pnpm install`, Sonar job은 `pnpm install --frozen-lockfile` 사용 |
| Frontend package    | `frontend/package.json`                  | `packageManager`가 `pnpm@10.33.0`으로 명시됨                                          |
| README              | `README.md`                              | 프론트엔드는 pnpm을 안내하지만 루트 패키지 매니저 정책은 별도 문서화되지 않음         |

## Design Diff

| 영역                 | As-is                                       | To-be                                                                    |
| -------------------- | ------------------------------------------- | ------------------------------------------------------------------------ |
| Root package manager | 명시 없음                                   | `packageManager`로 `pnpm@10.33.0` 명시                                   |
| Root install guard   | npm 설치 가능                               | npm의 package manager 검사와 preinstall guard로 pnpm이 아닌 install 차단 |
| Root lockfile        | `package-lock.json`과 `pnpm-lock.yaml` 혼재 | 루트 `pnpm-lock.yaml`만 유지                                             |
| Husky hooks          | `npx`로 실행                                | `pnpm exec`로 실행                                                       |
| CI frontend install  | job별 pnpm install 옵션 차이                | CI frontend install을 lockfile 고정 설치로 통일                          |
| 문서                 | 루트 명령 일부가 npm 기준                   | 루트와 frontend 모두 pnpm 기준임을 README/AGENTS에 명시                  |

## Requirements

- 루트 `package.json`은 공식 package manager를 pnpm으로 명시한다.
- 루트에서 `npm install`을 실행하면 package manager 정책 위반으로 실패한다.
- 루트 npm lockfile은 제거하고 루트 pnpm lockfile만 유지한다.
- Husky pre-commit과 commit-msg hook은 루트 pnpm 의존성을 기준으로 실행한다.
- README는 루트와 frontend Node.js 의존성 모두 pnpm을 사용한다고 안내한다.
- CI frontend install 명령은 lockfile 기준 설치를 사용한다.

## Acceptance Criteria

- 루트에서 어떤 package manager를 써야 하는지 `package.json`과 README로 확인할 수 있다.
- `package-lock.json`이 더 이상 추적되지 않는다.
- 루트 `pnpm install --lockfile-only`가 성공한다.
- `npm install --dry-run --ignore-scripts=false`가 실패하고 npm lockfile을 재생성하지 않는다.
- Husky 훅이 `npx` 대신 `pnpm exec`를 사용한다.

## Validation

- `pnpm install --lockfile-only`
- `npm install --dry-run --ignore-scripts=false`
- `pnpm exec lint-staged --version`
- `pnpm exec commitlint --version`
- `cd frontend && pnpm install --frozen-lockfile`
