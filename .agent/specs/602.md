# 로컬 개발 부트스트랩 원커맨드 제공

## Goal

신규 개발자가 루트의 단일 커맨드만으로 로컬 개발 스택 실행에 필요한 준비(`.env` 생성, backend JAR preflight, Docker Compose 설정 검증, 포트·필수 환경 변수 사전 점검)를 한 번에 마치고, 성공 시 접속 URL을 한 번에 확인할 수 있게 한다.

## Problem

현재 로컬 개발을 시작하려면 `cp .env.example .env`, backend `bootJar`, Docker Compose 실행 순서를 사람이 기억해야 한다. `.env`가 없으면 `env:check`가 수동 복사를 안내하며 중단되고, `.env.example`의 `JWT_SECRET`은 `<...>` placeholder라서 복사 후에도 수동 편집이 필요하다. Docker Compose 설정 오류나 포트 충돌은 `docker compose up` 단계에서야 드러나며, 필수 환경 변수(`JWT_SECRET`, `AIRFLOW_WEBHOOK_SECRET`, `DB_PASSWORD`)가 비어 있을 때 어떤 값이 왜 필요한지 즉시 알기 어렵다.

## Scope

- `scripts/dev-setup.mjs` 부트스트랩 스크립트를 추가한다.
  - `.env`가 없으면 `.env.example` 기반으로 생성한다. 생성 시 `JWT_SECRET` placeholder는 로컬 전용 랜덤 값으로 치환하고, 생성 사실과 검토 안내를 출력한다. 기존 `.env`는 수정하지 않는다.
  - Docker / Docker Compose 가용성을 확인한다.
  - 필수 환경 변수(`DB_PASSWORD`, `JWT_SECRET`, `AIRFLOW_WEBHOOK_SECRET`)의 누락·빈 값·`<...>` placeholder를 점검하고, 실패 시 어떤 값이 필요한지와 조치 방법을 출력한 뒤 종료 코드 1로 끝낸다.
  - backend JAR 준비는 기존 `scripts/preflight-backend-jar.mjs`의 `main`을 재사용한다.
  - `docker compose config --quiet`(해당 profile 기준)로 Compose 설정을 검증하고, 실패 시 compose 오류를 그대로 표출한다.
  - 호스트 포트(기본 5432/8080/5173/9000/9001, `*_HOST_PORT` 오버라이드 반영)를 사전 점검한다. 이미 실행 중인 본 프로젝트 compose 서비스는 정상으로 안내하고, 외부 프로세스 점유는 경고와 함께 README 포트 충돌 대응을 안내한다(경고는 실패가 아님).
  - `--print-urls` 플래그는 준비 단계를 건너뛰고 접속 URL 표만 출력한다(`--profile full`이면 Airflow URL 포함).
- 루트 `package.json`에 `dev:setup` script를 추가하고, `up:light`/`up:full`이 `env:check + preflight` 대신 부트스트랩을 수행한 뒤 기동하고 성공 시 접속 URL을 출력하도록 위임을 교체한다.
- `scripts/dev-setup.test.mjs`에 순수 함수 단위 테스트를 추가한다(CI `scripts-test` job이 자동 수집).
- README 시작하기 섹션과 AGENTS Docker Compose/Root Scripts 안내를 단일 커맨드 기준으로 갱신한다.

## Non-goals

- Makefile, justfile 등 별도 태스크 러너를 도입하지 않는다.
- backend/frontend/ML 애플리케이션 코드, API 계약, database schema, CI workflow를 변경하지 않는다.
- `up:pipeline`, `smoke`, `logs`, `codegen` 등 기존 `env:check` 사용 경로의 동작은 변경하지 않는다(`env:check` script 유지).
- `.env.example`의 키 구성과 placeholder 정책을 변경하지 않는다.
- 이미 존재하는 `.env`의 값을 자동 수정하지 않는다(점검·안내만 수행).
- 실서비스 secret 발급·주입(AWS Secrets Manager 등) 흐름은 다루지 않는다.

## Affected Paths

- `scripts/dev-setup.mjs` (신규)
- `scripts/dev-setup.test.mjs` (신규)
- `package.json`
- `README.md`
- `AGENTS.md`
- `.agent/specs/602.md`

## Requirements

1. 신규 개발자가 clone 후 `pnpm install && pnpm run dev`만으로 light 스택 준비와 기동을 완료할 수 있어야 한다(`.env` 수동 복사·편집 불필요).
2. `pnpm run dev:setup` 단독 실행은 기동 없이 준비·점검만 수행하고, 완료 시 다음 단계(`pnpm run dev`)를 안내해야 한다.
3. `.env` 생성 시 `JWT_SECRET`은 32바이트 이상 랜덤 값으로 채워져야 하며, 생성된 `.env`는 커밋 대상이 아니어야 한다(기존 `.gitignore` 유지).
4. 필수 환경 변수 점검 실패 메시지는 변수 이름, 실패 사유(누락/빈 값/placeholder), 조치 방법을 포함해야 한다.
5. Compose 설정 검증 실패와 JAR 빌드 실패는 종료 코드 1로 전파되어 `up:light`/`up:full`이 기동 단계로 진행하지 않아야 한다.
6. 포트 점검은 실패가 아닌 경고로 처리하되, 어떤 포트가 어떤 서비스용인지와 대응 방법이 로그에 남아야 한다.
7. 기동 성공 후 Frontend(5173)/Backend(8080)/MinIO 콘솔(9001) URL이 출력되어야 하며, full 모드에서는 Airflow(8081)도 포함해야 한다.
8. 순수 함수(env 파싱, placeholder 판정, 필수 변수 점검, URL 구성)는 Docker 없이 `node --test`로 검증 가능해야 한다.

## Data/API Impact

- API, persistence, runtime configuration 변경은 없다. Docker Compose 서비스 정의(`docker-compose.yml`)도 변경하지 않는다.
- 로컬 `.env` 파일이 스크립트로 생성될 수 있다는 점이 유일한 로컬 상태 변화다.

## Acceptance Criteria

- `.env`가 없는 상태에서 `pnpm run dev:setup`을 실행하면 `.env`가 생성되고 `JWT_SECRET`이 랜덤 값으로 채워지며, 준비 점검이 통과한 뒤 다음 단계 안내가 출력된다.
- `.env`가 있는 상태에서 재실행하면 `.env`를 덮어쓰지 않고 점검만 수행한다(멱등).
- `.env`에서 `JWT_SECRET`을 비우거나 placeholder로 두면 변수명·사유·조치 방법이 출력되고 종료 코드 1로 끝난다.
- `pnpm run dev`(=`up:light`)는 부트스트랩 통과 후 light 스택을 기동하고 접속 URL 표를 출력한다.
- `node --test "scripts/**/*.test.mjs"`가 신규 테스트를 포함해 통과한다.
- README 시작하기 섹션이 단일 커맨드 흐름을 안내한다.

## Validation

- `node --test "scripts/**/*.test.mjs"`
- `node -e 'JSON.parse(require("fs").readFileSync("package.json", "utf8")); console.log("package.json ok")'`
- `pnpm run format:check`
- `pnpm exec prettier --check .agent/specs/602.md AGENTS.md scripts/dev-setup.mjs scripts/dev-setup.test.mjs package.json` (README.md는 main 기준으로 이미 prettier 미준수 상태라 전체 파일 검사 대신 변경 구간만 기존 표기 스타일을 따른다)
- Docker 가용 환경에서 `.env` 백업 후 `pnpm run dev:setup` 실제 실행(생성·멱등·실패 경로)

## Open Questions

- 없음. 이슈 본문이 부트스트랩 단계(env 생성, JAR 확인, compose 검증, 포트·env 점검, URL 출력)를 명시하고 있어 범위 결정에 충분하다. `JWT_SECRET` 자동 생성은 "단일 커맨드만으로 준비" 확인 포인트를 만족하기 위한 선택이며, 로컬 전용 값이라 보안 영향이 없다.
