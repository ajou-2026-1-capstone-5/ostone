# 327: 상담 기록 화면의 워크스페이스 스코프와 URL 선택 상태 정리

## Goal

상담 기록 화면이 워크스페이스별 상담 세션만 조회하고, 선택한 세션을 URL 상태와 일관되게 동기화한다.

---

## Background

상담 기록 화면은 `/workspaces/:workspaceId/consultation/history` 하위 라우트에 있지만 세션 목록 API와 선택 상태가 워크스페이스 경계를 충분히 반영하지 않았다. 상담 로그는 운영 데이터이므로 다른 워크스페이스의 세션이 섞이지 않아야 하고, 공유 가능한 URL로 동일한 선택 상태를 복원할 수 있어야 한다.

---

## REST API

| Method | Path | Description |
| --- | --- | --- |
| `GET` | `/api/v1/workspaces/{workspaceId}/consultation/sessions` | 워크스페이스 상담 세션 목록 조회 |

### Query Parameters

| Name | Type | Required | Description |
| --- | --- | --- | --- |
| `status` | string | no | 세션 상태 필터 |
| `page` | number | no | 페이지 번호, 기본값 `0` |
| `size` | number | no | 페이지 크기, 기본값 `20`, 최대 `100` |

### Response

기존 `CounselorSessionResponse` 형태를 유지한다.

```json
{
  "content": [
    {
      "id": 7,
      "status": "COMPLETED",
      "channel": "WEB",
      "metaJson": "{}",
      "startedAt": "2026-05-22T09:00:00+09:00"
    }
  ],
  "page": 0,
  "size": 20,
  "totalElements": 1,
  "totalPages": 1
}
```

## Backend Design

### Controller

`CounselorSessionController#getSessions`는 `workspaceId` path variable을 받는다.

```text
GET /api/v1/workspaces/{workspaceId}/consultation/sessions
```

기존 assign/release endpoint는 `/api/v1/consultation/sessions/{sessionId}/...` 경로를 유지한다.

### Application Service

`CounselorService#getSessions`는 다음 조건으로만 세션을 조회한다.

1. `workspaceId`는 양수여야 한다.
2. `status`가 있으면 `ChatSessionStatus`로 검증한다.
3. `page >= 0`, `size > 0` 조건을 검증한다.
4. 상태 필터가 있으면 `workspaceId + status + pageable`로 조회한다.
5. 상태 필터가 없으면 `workspaceId + pageable`로 조회한다.

### Repository

`ChatSessionRepository`에 워크스페이스 기반 page 조회 메서드를 추가한다.

```java
Page<ChatSession> findByWorkspaceId(Long workspaceId, Pageable pageable);

Page<ChatSession> findByWorkspaceIdAndStatus(
    Long workspaceId, ChatSessionStatus status, Pageable pageable);
```

## Frontend Design

### API Integration

`consultationApi.getSessions`는 `workspaceId`를 필수 인자로 받아 workspace-scoped URL을 호출한다.

```typescript
consultationApi.getSessions(workspaceId, { status, page, size });
```

`useChatSessions`는 `workspaceId`를 query key뿐 아니라 실제 요청에도 전달한다. 유효한 `workspaceId`가 없으면 query를 실행하지 않는다.

### Route State

상담 기록 화면은 다음 라우트를 지원한다.

```text
/workspaces/:workspaceId/consultation/history
/workspaces/:workspaceId/consultation/history/:sessionId
```

세션 카드를 선택하면 `navigate`로 `sessionId`가 포함된 URL로 이동한다.

### Invalid Session State

URL의 `sessionId`가 현재 워크스페이스의 세션 목록에 없으면 메시지 조회를 실행하지 않고 명확한 안내 상태를 표시한다.

## Test Plan

### Backend

- `CounselorServiceTest`
  - 상태 필터 없이 `workspaceId`로 세션 페이지를 조회한다.
  - 상태 필터가 있으면 `workspaceId + status`로 세션 페이지를 조회한다.
  - 유효하지 않은 status는 `BadRequestException`을 반환한다.
  - 유효하지 않은 workspaceId는 `BadRequestException`을 반환한다.
- `CounselorSessionControllerTest`
  - `GET /api/v1/workspaces/{workspaceId}/consultation/sessions` 요청을 service에 위임한다.
  - status/page/size query parameter를 service에 전달한다.

### Frontend

- `consultationApi.test.ts`
  - 세션 목록 요청이 `/api/v1/workspaces/{workspaceId}/consultation/sessions`를 호출한다.
  - status/page/size query parameter를 보존한다.
- `chatHistoryApi.test.ts`
  - `workspaceId`를 실제 API 요청에 전달한다.
  - `workspaceId`가 없으면 query를 실행하지 않는다.
- `ChatHistoryPage.test.tsx`
  - URL의 workspaceId로 완료 세션 목록을 조회한다.
  - 세션 선택 시 URL의 `sessionId`가 갱신된다.
  - URL의 `sessionId`로 직접 진입하면 해당 세션 메시지를 조회한다.
  - 현재 워크스페이스에 없는 `sessionId`는 메시지를 조회하지 않고 안내 상태를 표시한다.
