# 자동화 커버리지 지표

## Goal

워크스페이스 대시보드에서 운영 지식팩이 실제 상담을 얼마나 자동화하고 있는지 workflow 매칭, intent 분류, 자동 완료, 상담사 개입, 저신뢰/미매칭, LLM-only 처리 비율, 기간별 추이로 확인할 수 있게 한다.

## 범위

- `backend/src/main/java/com/init/workflowruntime/presentation/ConsultationMetricsController.java`
  - 기존 `GET /api/v1/workspaces/{workspaceId}/consultation/metrics` 응답에 자동화 커버리지 지표를 포함한다.
  - 기존 workspace membership 검증과 기간 쿼리 파라미터를 유지한다.
- `backend/src/main/java/com/init/workflowruntime/application/ConsultationMetricsService.java`
  - 선택 기간의 운영 상담 세션 기준으로 커버리지 지표를 계산한다.
  - 지표 산출 로그가 부족하면 `NEEDS_INSTRUMENTATION` 상태를 응답한다.
- `backend/src/main/java/com/init/workflowruntime/infrastructure/persistence/JpaConsultationMetricsRepository.java`
  - `runtime.chat_session`, `runtime.workflow_execution`, `runtime.decision_log`, `runtime.workflow_match_decision` 기반 세션 fact를 확장한다.
  - simulation/demo 채널 세션은 운영 커버리지에서 제외한다.
- `frontend/src/features/consultation/api/consultationApi.ts`
  - 상담 지표 타입에 커버리지 응답 구조를 추가한다.
- `frontend/src/pages/workspace/ui/WorkspaceDashboardPage.tsx`
  - #517/#518 대시보드 셸과 기간 필터 기준에 맞춰 자동화 커버리지 카드와 기간별 추이를 표시한다.
  - 고객 화면 표현은 `Domain Pack` 대신 “운영 지식팩”을 사용한다.

## 제외 범위

- 추천 액션 생성
- 워크플로우 상세 병목 분석
- LLM 기반 진단
- 저신뢰/미매칭 로그 자체의 신규 계측 보강
- domain pack version, channel, workflow status 공통 필터의 서버 집계 반영

## API

### `GET /api/v1/workspaces/{workspaceId}/consultation/metrics`

기존 응답에 `coverage` 필드를 추가한다.

| 필드 | 설명 |
| --- | --- |
| `workflowMatchedCount` | 선택 기간 운영 상담 중 workflow가 매칭된 세션 수 |
| `workflowMatchRate` | 전체 운영 상담 중 workflow 매칭 비율 |
| `intentClassificationSuccessCount` | intent가 성공적으로 선택/매칭된 세션 수 |
| `intentClassificationSuccessRate` | 전체 운영 상담 중 intent 분류 성공 비율 |
| `lowConfidenceCount` | 저신뢰 결정으로 분류된 세션 수 |
| `lowConfidenceRate` | 전체 운영 상담 중 저신뢰 비율 |
| `unmatchedSessionCount` | workflow/intent를 매칭하지 못한 세션 수 |
| `autoCompletedWorkflowCount` | workflow 매칭 후 상담사 개입 없이 자동 완료된 세션 수 |
| `humanHandoffRate` | 전체 운영 상담 중 상담사 개입 비율 |
| `llmOnlyProcessingRate` | 완료 상담 중 LLM-only 처리 비율 |
| `measurementStatus` | `READY` 또는 `NEEDS_INSTRUMENTATION` |
| `measurementMessage` | 계측 상태 설명 |
| `trend` | 기간별 커버리지 추이 |

### Trend Point

| 필드 | 설명 |
| --- | --- |
| `date` | `yyyy-MM-dd` 기준일 |
| `totalConsultationCount` | 해당 일자에 시작된 운영 상담 수 |
| `workflowMatchedCount` | 해당 일자 workflow 매칭 세션 수 |
| `workflowMatchRate` | 해당 일자 workflow 매칭 비율 |

## 데이터 기준

- 운영 상담 모집단은 선택 기간에 시작된 `runtime.chat_session`이며, #518과 동일하게 demo/simulation 채널은 제외한다.
- workflow 매칭 성공은 `runtime.workflow_execution.workflow_definition_id` 또는 `runtime.workflow_match_decision.selected_workflow_id`가 있는 세션으로 본다.
- intent 분류 성공은 `runtime.workflow_execution.intent_definition_id`, `runtime.workflow_match_decision.selected_intent_id`, 또는 `runtime.decision_log.decision_type = 'INTENT_SELECTED'`로 본다.
- 저신뢰는 `runtime.decision_log.confidence_score < 0.7` 또는 `runtime.workflow_match_decision.status in ('AMBIGUOUS', 'BLOCKED', 'ERROR')`인 세션으로 본다.
- 미매칭은 `runtime.workflow_match_decision.status = 'UNKNOWN'`이거나 커버리지 로그가 있는데 workflow/intent가 모두 없는 세션으로 본다.
- 자동 완료는 workflow 매칭 세션 중 선택 기간에 완료 처리됐고 상담사/상담원 메시지가 없는 세션으로 본다.
- 상담사 개입은 기존 상담 처리 지표와 동일하게 상담사/상담원 메시지 기반으로 계산한다.
- LLM-only 처리 비율은 완료 상담 중 상담사 메시지 없이 LLM 메시지가 있는 세션 비율로 계산한다.
- 선택 기간에 운영 상담이 있는데 `decision_log`와 `workflow_match_decision` 계측 로그가 모두 없으면 `NEEDS_INSTRUMENTATION`을 반환한다.

## UI 요구사항

- 대시보드에 “자동화 커버리지” 영역을 추가한다.
- 전체 상담 중 workflow 매칭 비율을 표시한다.
- 자동 완료와 상담사 개입을 분리해 표시한다.
- 저신뢰/미매칭 케이스 수를 표시한다.
- LLM-only 처리 비율을 표시한다.
- 기간별 커버리지 추이를 표시한다.
- 계측 로그가 부족하면 “계측 필요” 상태와 설명을 표시한다.
- 평균/비율 계산이 불가능한 경우 `--`로 표시한다.

## Acceptance Criteria

1. 전체 상담 중 workflow에 매칭된 비율이 표시된다.
2. 자동 완료와 상담사 개입이 분리된다.
3. 저신뢰/미매칭 케이스 수가 표시된다.
4. LLM-only 처리 비율이 표시된다.
5. 기간별 커버리지 추이가 표시된다.
6. 지표 산출에 필요한 로그가 부족하면 “계측 필요” 상태가 표시된다.
7. simulation/demo 채널 세션은 자동화 커버리지에 포함되지 않는다.
8. 워크스페이스 멤버가 아닌 사용자의 API 요청은 거부된다.

## Validation

- Backend service test: workflow/intent 매칭률, 저신뢰/미매칭, 자동 완료, LLM-only 비율, 계측 필요 상태를 검증한다.
- Backend repository test: `runtime.decision_log`, `runtime.workflow_match_decision`, `runtime.workflow_execution` 기반 fact 확장과 simulation/demo 제외를 검증한다.
- Backend controller test: `coverage` 응답 필드를 검증한다.
- Frontend API test: 커버리지 응답 타입을 포함한 metrics 호출을 검증한다.
- Frontend page test: 자동화 커버리지 카드, 추이, 계측 필요 상태를 검증한다.

## Open Questions

- 저신뢰 기준값 `0.7`은 현재 구현의 1차 기준으로 사용하며, 운영 임계값이 정해지면 별도 설정화할 수 있다.
- 저신뢰/미매칭 로그 보강 자체는 이 이슈 범위 밖이며, 부족한 경우 하위 계측 작업으로 분리한다.
